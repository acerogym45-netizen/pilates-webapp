
### 콘솔 로그 분석

사용자가 제공한 스크린샷에서 다음 로그를 발견:
```
📅 Loading curriculum for 2026-4, complex: a3bd3a67-f7e0-4edb-a9bb-236347e0d5e4
✅ Fetched 1 total curriculums
Checking curriculum: complex=cheongju-sk, year=2026, month=4, active=true
❌ Target curriculum found: undefined
```

**문제 발견**:
- 조회 시 사용하는 complex_id: `a3bd3a67-f7e0-4edb-a9bb-236347e0d5e4` (UUID)
- 저장된 커리큘럼의 complex_id: `cheongju-sk` (코드명)
- **불일치** → 매칭 실패 → `undefined` 반환

## 🔍 원인 분석

### 시스템 구조

프로젝트에는 **2가지 단지 식별자**가 있습니다:

1. **UUID (id)**: 데이터베이스의 고유 식별자
   - 예: `a3bd3a67-f7e0-4edb-a9bb-236347e0d5e4`
   - `complex_settings` 테이블의 `id` 필드

2. **Complex Code (complex_code)**: 사용자 친화적인 코드명
   - 예: `cheongju-sk`, `sejong-lake`
   - `complex_settings` 테이블의 `complex_code` 필드

### 문제의 근본 원인

#### 입주자 페이지 (`js/main.js`)
```javascript
// 1549줄
const complexId = complexContext.getComplexId();  // ← UUID 반환
```

#### ComplexContext 클래스 (`js/complex-context.js`)
```javascript
// 70~72줄
getComplexId() {
    return this.currentComplex ? this.currentComplex.id : null;  // ← UUID 반환
}

// 75~77줄
getComplexCode() {
    return this.currentComplex ? this.currentComplex.complex_code : null;  // ← Code 반환
}
```

#### 관리자 페이지 (`js/admin-notices.js`)
```javascript
// 442줄
const curriculumData = {
    complex_id: currentComplexCode,  // ← 'cheongju-sk' 저장
    // ...
};
```

**결론**: 
- **입주자 페이지**: UUID로 조회
- **관리자 페이지**: complex_code로 저장
- **결과**: 영원히 매칭 불가능 ❌

## ✅ 해결 방법

### 수정 코드

#### ❌ 이전 코드 (`js/main.js` 1549줄)
```javascript
const complexId = complexContext.getComplexId();  // UUID 반환
```

#### ✅ 수정된 코드
```javascript
const complexId = complexContext.getComplexCode();  // complex_code 반환
```

### 수정 사유

**왜 UUID가 아닌 complex_code를 사용?**

1. **일관성**: 관리자 페이지가 이미 `complex_code`로 저장하고 있음
2. **가독성**: `cheongju-sk` > `a3bd3a67-f7e0-4edb-a9bb-236347e0d5e4`
3. **유지보수**: URL 파라미터와 동일 (`?complex=cheongju-sk`)
4. **호환성**: 기존 데이터와 호환

**대안: 관리자 페이지를 UUID로 변경?**
- ❌ 이미 저장된 모든 커리큘럼 데이터 수정 필요
- ❌ 데이터 마이그레이션 리스크
- ✅ 입주자 페이지만 수정하는 것이 안전

**권장 개선 방안**:

#### 옵션 A: 모두 complex_code로 통일 (권장)

- 가독성 높음
- URL 파라미터와 일치
- 데이터 조회 시 추가 JOIN 불필요

**단점**:
- 기존 데이터 마이그레이션 필요 (신규 프로젝트에 권장)

#### 옵션 B: 모두 UUID로 통일
```javascript
// 모든 테이블에서

```

**장점**:
- 데이터베이스 정규화
- 외래 키 무결성

**단점**:
- 가독성 낮음
- URL에 UUID 노출

#### 옵션 C: 현재 상태 유지 (현재 선택)
```javascript
// 혼용 (테이블마다 다름)

```

**장점**:
- 기존 데이터 수정 불필요
- 배포 리스크 0%

**단점**:
- 혼란 가능성

**결론**: 현재는 옵션 C로 유지하되, 향후 대규모 리팩토링 시 옵션 A 권장
